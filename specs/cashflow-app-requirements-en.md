# CashFlow — Product Requirements Document

**Version:** 1.1  
**Date:** April 2026  
**Stack:** Kotlin Multiplatform (KMP) + Compose Multiplatform  
**Package:** `com.github.dsbezerra.cashflow`  
**Platform Priority:** Desktop (JVM) → Android → iOS  
**Development OS:** Windows (Android Studio)  
**Language:** Portuguese (pt-BR)

---

## 1. Overview

A full-featured personal cash flow app built with Kotlin Multiplatform and Compose Multiplatform. Business logic, persistence, and navigation live in `commonMain` and are shared across all targets. The UI is implemented entirely in Compose Multiplatform. The app is offline-first with an architecture prepared for cloud sync in future versions.

---

## 2. Project Structure

### 2.1 Current Setup (single-module)

The project was generated by Android Studio's KMP wizard. All source lives inside a single `composeApp` module with the following sourceSets:

```
composeApp/
├── commonMain/         # Shared logic, UI, models, repositories, use cases
├── commonTest/         # Shared unit tests
├── androidMain/        # Android-specific drivers and entry point
├── iosMain/            # iOS-specific drivers (build via macOS later)
├── jvmMain/            # Desktop (JVM) entry point and drivers
└── ...
```

> No `:shared` module separation for now. If the project grows significantly,
> extracting a `:shared` module is recommended before v2.0.

## 3. Architecture

### 3.1 Architectural Pattern

- **MVI (Model-View-Intent)** for screen state management
- **Clean Architecture** in layers within `commonMain`:
  ```
  domain/       → entities, use cases, repository interfaces
  data/         → repository implementations, database, mappers
  presentation/ → screens, ViewModels, UI state
  di/           → Koin modules
  navigation/   → NavGraph, route definitions
  ```

### 3.2 Technology Stack

| Layer | Library | Version |
|---|---|---|
| UI | Compose Multiplatform | 1.7.0 |
| Navigation | Jetpack Navigation Compose (multiplatform) | 2.8.x |
| ViewModel | `androidx.lifecycle:lifecycle-viewmodel-compose` | 2.8.4 |
| Database | SQLDelight | 2.0.2 |
| DI | Koin Multiplatform | 4.0.0 |
| Async | Kotlinx Coroutines | 1.9.0 |
| Serialization | Kotlinx Serialization JSON | 1.7.3 |
| Date / Time | Kotlinx DateTime | 0.6.1 |
| Charts | Compose Canvas (custom) or Koalaplot | — |
| Testing | kotlin.test + Turbine | — |

> **Note:** `androidx.navigation:navigation-compose` has multiplatform support
> since 2.8. Type-safe routes via `@Serializable` data classes (Kotlinx
> Serialization) are the recommended pattern and are used throughout this project.

### 3.3 Platform Drivers

| Concern | Desktop (JVM) | Android | iOS |
|---|---|---|---|
| SQLDelight driver | `sqlite-driver` | `android-driver` | `native-driver` |
| Coroutines dispatcher | `kotlinx-coroutines-swing` | Default | Default |
| File system (exports) | `java.io.File` | Android `Context` | `NSFileManager` |

## 4. Domain

### 4.1 Core Entities

#### `Account`
```kotlin
id: String              // UUID string
name: String
type: AccountType       // CHECKING, SAVINGS, CASH, CREDIT, INVESTMENT
currency: String        // "BRL", "USD", etc.
initialBalance: Double
color: String           // hex color for visual identification
icon: String            // icon identifier key
isArchived: Boolean
createdAt: Long         // epoch millis (kotlinx-datetime)
```

#### `Transaction`
```kotlin
id: String
accountId: String
categoryId: String
type: TransactionType   // INCOME, EXPENSE, TRANSFER
amount: Double
description: String
date: Long              // epoch millis
notes: String?
attachmentPath: String? // local file path for receipt image
isRecurring: Boolean
recurringId: String?    // FK to RecurringRule
createdAt: Long
updatedAt: Long
```

#### `Category`
```kotlin
id: String
name: String
type: CategoryType      // INCOME, EXPENSE, BOTH
icon: String
color: String
parentId: String?       // subcategory support (1 level deep)
isDefault: Boolean
isArchived: Boolean
```

#### `RecurringRule`
```kotlin
id: String
accountId: String
categoryId: String
type: TransactionType
amount: Double
description: String
frequency: Frequency    // DAILY, WEEKLY, MONTHLY, YEARLY
interval: Int           // e.g. 2 = every 2 weeks
startDate: Long
endDate: Long?
nextOccurrence: Long
isActive: Boolean
```

#### `Budget`
```kotlin
id: String
categoryId: String
amount: Double
period: BudgetPeriod    // MONTHLY, YEARLY
startDate: Long
isActive: Boolean
```

> All IDs are UUID strings generated in `commonMain` to ensure future
> sync compatibility. Dates are stored as epoch millis (Long) for
> SQLDelight cross-platform compatibility.

## 5. Features

### 5.1 Account Management

- [ ] Balance evolution over time (line chart)
- [ ] Spending by category (horizontal bar chart)

## 6. Navigation

Navigation uses **Jetpack Navigation Compose** (`androidx.navigation:navigation-compose`) in `commonMain` with **type-safe routes** defined as `@Serializable` data classes.

### 6.1 Route Definitions

```kotlin
// commonMain/navigation/Routes.kt
sealed interface Route {

    @Serializable object Dashboard : Route

    @Serializable object TransactionList : Route
    @Serializable data class TransactionDetail(val id: String? = null) : Route  // null = create

    @Serializable object ReportByPeriod : Route
    @Serializable object ReportByCategory : Route

    @Serializable object AccountList : Route
    @Serializable data class AccountDetail(val id: String) : Route
    @Serializable data class AccountForm(val id: String? = null) : Route        // null = create

    @Serializable object CategoryList : Route
    @Serializable object BudgetList : Route
    @Serializable object RecurringRuleList : Route
    @Serializable object Export : Route
    @Serializable object About : Route
}
```

### 6.2 Nav Graph Structure

```
NavHost (root)
├── Dashboard
├── TransactionList
│   └── → TransactionDetail(id?)
├── ReportByPeriod
├── ReportByCategory
├── AccountList
│   ├── → AccountDetail(id)
│   └── → AccountForm(id?)
├── CategoryList
├── BudgetList
├── RecurringRuleList
├── Export
└── About
```

### 6.3 App Shell

The outer shell composable (`AppShell`) wraps the `NavHost` and renders the platform-appropriate navigation chrome:

| Platform | Navigation chrome |
|---|---|
| Desktop (JVM) | `NavigationRail` (left side) |
| Android | `NavigationBar` (bottom) |

The FAB (new transaction) is rendered in `AppShell` and always navigates to `TransactionDetail(id = null)`.

## 7. Persistence

### 7.1 SQLDelight Setup

```
# .sq files location
commonMain/sqldelight/com/github/dsbezerra/cashflow/db/
├── Account.sq
├── Transaction.sq
├── Category.sq
├── RecurringRule.sq
├── Budget.sq
└── migrations/
    └── 1.sqm
```

Generated code package: `com.github.dsbezerra.cashflow.db`  
Database name: `CashFlowDatabase`

### 7.2 Driver Factory (expect/actual)

```kotlin
// commonMain
expect class DriverFactory {
    fun createDriver(): SqlDriver
}

// jvmMain (Desktop)
actual class DriverFactory {
    actual fun createDriver() = JdbcSqliteDriver(
        "jdbc:sqlite:cashflow.db",
        schema = CashFlowDatabase.Schema
    )
}

// androidMain
actual class DriverFactory(private val context: Context) {
    actual fun createDriver() = AndroidSqliteDriver(
        CashFlowDatabase.Schema, context, "cashflow.db"
    )
}

// iosMain
actual class DriverFactory {
    actual fun createDriver() = NativeSqliteDriver(
        CashFlowDatabase.Schema, "cashflow.db"
    )
}
```

### 7.3 Reactive Queries

All list queries return `Flow<List<T>>` via SQLDelight's coroutines extension, enabling automatic UI recomposition when data changes.

### 7.4 Data Seeding

On first launch (`PRAGMA user_version = 0`), populate default pt-BR categories with icons and colors. No sample transactions — real data from day one.

### 7.5 Sync Readiness (Future)

- All IDs are UUID strings generated locally in `commonMain`
- `createdAt` / `updatedAt` on every entity
- `syncStatus` column reserved: `SYNCED | PENDING | CONFLICT`
- Repositories are interfaces in `domain/` — implementations are swappable

## 8. Dependency Injection (Koin)

```kotlin
// commonMain/di/
val databaseModule = module { ... }
val repositoryModule = module { ... }
val useCaseModule = module { ... }
val viewModelModule = module { ... }    // viewModel { } blocks for Jetpack ViewModels

// Platform-specific (expect/actual)
expect val platformModule: Module       // provides DriverFactory per target
```

Koin is started in each platform entry point:

```kotlin
// jvmMain - Main.kt
startKoin {
    modules(platformModule, databaseModule, repositoryModule, useCaseModule, viewModelModule)
}

// androidMain - MainApplication.kt (Application subclass)
startKoin {
    androidContext(this@MainApplication)
    modules(platformModule, databaseModule, repositoryModule, useCaseModule, viewModelModule)
}
```

## 9. UX & Design

### 9.1 Visual Guidelines

- Material 3 design system (`compose.material3`) — already in `commonMain.dependencies`
- Light and dark mode via `MaterialTheme` (follows system preference)
- Green for income, red for expenses, gray for transfers
- Consistent color tokens per account and category

### 9.2 Desktop Specifics

- `NavigationRail` on the left side
- Max-width constraint on content (~800dp centered)
- Keyboard shortcut: `Ctrl+N` → new transaction
- Window minimum size: 900×600px

### 9.3 Android Specifics

- `NavigationBar` at the bottom
- Edge-to-edge layout (`WindowCompat.setDecorFitsSystemWindows`)
- FAB for new transaction

### 9.4 Behavior

- Confirmation dialog before deleting a transaction or account
- Automatic currency formatting (R$ 1.234,56) while typing
- Pull-to-refresh on transaction list (Android)
- Lazy pagination for large transaction lists

## 11. Non-Functional Requirements

- **Performance:** Smooth scrolling with 10,000+ transactions (lazy pagination)
- **Offline-first:** 100% functional with no network connection
- **Desktop startup:** Cold start < 2s on a mid-range Windows PC
- **Android compatibility:** minSdk 26 (Android 8.0+)
- **iOS compatibility:** iOS 16+ (deferred phase)
